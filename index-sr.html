Kubernetes

Arhitektura, primene i put od VM-ova do kontejnera

K8s = „K” + 8 slova + „s”

Deo 1: Istorijske potrebe

Od jedne aplikacije po serveru do orkestracije hiljada kontejnera

Fizički serveri (početak 2000-ih)

  • Jedna aplikacija (ili nekoliko) po mašini
  • Teško skaliranje: kupovina novog hardvera, ručna konfiguracija
  • Nedovoljno iskorišćen kapacitet: CPU i RAM često u praznom hodu
  • Sporo nabavljane: dani ili nedelje za novi server

Potreba: bolje iskorišćenje resursa i brže nabavljane

Virtualizacija (VM-ovi)

  • Hipervizor (VMware, KVM, Hyper-V) pokreće više VM-ova na jednom hostu
  • Svaki VM: pun OS (kernel, biblioteke, aplikacija) — jaka izolacija
  • Prednosti: konsolidacija, snapshot-i, migracija, multi-tenancy
  • Mana: teški — svaki VM je ceo OS, spor podizanje i velika potrošnja resursa

Potreba: lakše i brže jedinice za deploy

Kontejneri (2010-e)

  • Kontejneri dele kernel hosta; izoluju procese, fajl sistem, mrežu
  • Docker (2013) je uveo image-e i alate u masovnu upotrebu: jednom napraviš, pokrećeš svuda
  • Znatno lakši od VM-ova: brzo podizanje, manje overhead-a, veća gustina
  • Rezultat: mnogo malih kontejnera po hostu → potreba za upravljanjem i raspoređivanjem

Potreba: orkestracija u velikoj meri

Od „pokreni kontejner” do „pokreni sistem”

  • Stotine ili hiljade kontejnera na mnogo mašina
  • Ko ih pokreće? Gde? Kako se međusobno pronalaze? Ko ih ponovo pokreće?
  • Potreba: orkestracija — raspoređivanje, mreža, skladištenje, skaliranje, rolling update-ovi

To je problem koji Kubernetes rešava.

Šta je Kubernetes?

„Kubernetes je open-source sistem za automatizaciju deploy-a, skaliranja i upravljanja kontejnerizovanim aplikacijama.”
  • Nastao iz Google-ovog Borg-a / Omega-e; open-source od 2014. (v1.0 2015.)
  • Dan pod CNCF-om; podrazumevani izbor za orkestraciju kontejnera
  • Deklarativan: opisuješ željeno stanje; K8s teži da cluster to stanje ostvari

Zašto koristiti Kubernetes?

  • Prenosivost — isti API on-prem, AWS, GCP, Azure, edge
  • Skaliranje — skaliraj aplikacije (i cluster) gore/dole
  • Otpornost — ponovo pokreće pala kontejnere, zamenjuje čvorove
  • Rolling update-i i rollback — deploy bez zastoja
  • Ekosistem — Helm, operatori, service mesh-evi, alati za observability

Arhitektura: visoki nivo

Dve strane: kontrolna ravan (mozak) i radni čvorovi (pokreću tvoje radno opterećenje)

Kontrolna ravan

  • API server
  • Scheduler
  • Controller manager
  • etcd (skladište stanja)

Radni čvorovi

  • kubelet
  • kube-proxy
  • Container runtime (containerd, CRI-O)

Komponente kontrolne ravni

  • API server — jedina ulazna tačka; REST API; validira i perzistira stanje
  • Scheduler — dodeljuje nove Pod-ove čvorovima (resursi, ograničenja)
  • Controller manager — mnogo kontrolera (Deployment, ReplicaSet, Node, …) koji posmatraju stanje i usklađuju (npr. „hoću 3 replike” → pokretanje/zaustavljanje Pod-ova)
  • etcd — distribuirana key-value baza; izvor istine za stanje cluster-a

Komponente radnog čvora

  • kubelet — agent na svakom čvoru; osigurava da kontejneri rade kako kontrolna ravan određuje
  • kube-proxy — mrežna pravila da Service-i (cluster IP, load balancing) rade
  • Container runtime — pokreće kontejnere (containerd, CRI-O); implementira CRI (Container Runtime Interface)

Tvoji Pod-ovi (kontejneri) rade na radnim čvorovima.

Ključni koncepti (gradivni blokovi)

  • Pod — najmanja jedinica za deploy; jedan ili više kontejnera koji dele mrežu/skladište
  • Deployment — deklarativni update-i za Pod-ove (replike, rolling update, rollback)
  • Service — stabilni mrežni identitet i load balancing ka Pod-ovima (ClusterIP, NodePort, LoadBalancer)
  • ConfigMap / Secret — konfiguracija i osetljivi podaci
  • Namespace — logičko grupisanje i izolacija unutar cluster-a

Još koncepata

  • StatefulSet — za stateful radna opterećenja (stabilan identitet, uređen deploy/skala)
  • DaemonSet — jedan Pod po čvoru (npr. logovanje, monitoring)
  • Ingress — HTTP(S) rutiranje u cluster
  • HPA (Horizontal Pod Autoscaler) — skaliranje Pod-ova po CPU/memoriji ili custom metrikama
  • PersistentVolume (PV) / PVC — trajno skladište za Pod-ove

Primene

Kada i kako organizacije koriste Kubernetes

Mikroservisi i cloud-native aplikacije

Primena: Mnogo malih servisa (API-ji, worker-i, UI-ji) koji se nezavisno skaliraju i deploy-uju.
  • Svaki servis = Deployment + Service (često i ConfigMap/Secret)
  • Service discovery preko DNS-a (npr. my-svc.namespace.svc.cluster.local)
  • Rolling update po servisu bez gašenja celog sistema

CI/CD i GitOps

Primena: Push koda → pipeline gradi image-e → deploy u K8s (npr. Argo CD, Flux, Jenkins).
  • Kubernetes kao cilj za sve okruženja (dev/stage/prod)
  • Deklarativni manifesti u Git-u; alati usklađuju stanje cluster-a sa Git-om (GitOps)
  • Konzistentni, revidirani, ponovljivi deploy-i

Hibridni i multi-cloud

Primena: Ista aplikacija radi on-prem, u jednom ili više oblaka, ili na edge-u.
  • Kubernetes API je standardizovan; dobavljači nude managed K8s (EKS, GKE, AKS, OpenShift)
  • Manje vendor lock-in-a; iste veštine i alati u svim okruženjima

Batch poslovi i obrada podataka

Primena: Jednokratni ili zakazani poslovi (CronJob), ML trening, ETL.
  • Job — radi do završetka (npr. migracioni skript)
  • CronJob — radi po rasporedu
  • Cluster može automatski skalirati čvorove (npr. Karpenter, cluster autoscaler) za burst radna opterećenja

Visoka dostupnost i oporavak od katastrofe

  • Replike raspoređene na čvorove/zone; pad čvora ne obara aplikaciju
  • Više čvorova kontrolne ravni + etcd za HA
  • Backup/restore etcd-a i perzistentnih podataka; replikacija cluster-a između regiona za DR

Rezime

  • Istorija: Fizički serveri → VM-ovi (virtualizacija) → kontejneri (Docker) → potreba za orkestracijom → Kubernetes
  • K8s: Deklarativna, prenosiva platforma za pokretanje i upravljanje kontejnerizovanim radnim opterećenjima
  • Arhitektura: Kontrolna ravan (API, scheduler, kontroleri, etcd) + radnici (kubelet, kube-proxy, runtime)
  • Primene: Mikroservisi, CI/CD, hibridni/multi-cloud, batch/ML, HA/DR
